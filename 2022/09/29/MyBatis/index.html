MyBatis

1、MyBatis架构

（1）基础支撑层

  • 类型转换模块：提供类的别名机制、JDBC类型和Java类型之间的转换；在SQL绑定类型参数时，会将Java类型转换为JDBC类型，在ResultSet映射结果时，会将JDBC类型转换为Java类型；

  • 日志模块：MyBatis 提供了日志模块来集成 Java 生态中的第三方日志框架，该模块目前可以集成 Log4j、Log4j2、slf4j 等优秀的日志框架；

  • 反射工具模块：在Java反射包基础上进行封装，可以缓存反射相关元数据，提高反射效率；

  • Binding模块：通过SqlSession获取到Mapper接口，绑定对应的Mapper.xml，自动生成Mapper接口的代理对象，可以将错误提前到编译器；

  • 数据源模块：Mybatis默认实现一套数据源，同时也提供与第三方集成的接口；

  • 缓存模块：Mybatis自带一级、二级缓存，可以缓解数据访问压力；

  • 解析器模块：主要解析Mapper.xml和mybatis-config.xml；

  • 事务模块：Mybatis对数据库事务进行简单抽象，提供简单的接口和实现；

（2）核心处理层

  • 涉及MyBatis的初始化和一条SQL语句的执行流程；

  • 配置解析：Mybatis初始化过程中，会加载mybatis-config.xml、Mapper.xml和Mapper接口的信息，并将这些信息封装到Configuration对象中；如我们可以通过Configuration获取SqlSessionFactory对象，最终创建一个SqlSesseion对象操作数据库；

  • SQL解析和scripting模块：scripting是生成动态SQL的核心模块，他会根据运行时用户传入的参数，解析动态SQL标签，并生成SQL模板，然后处理SQL模板中的占位符，将传入的参数填写占位符，得到数据库真正执行的SQL语句；

  • SQL执行：Exceutor会调用事务模块实现事务相关控制，通过缓存模块实现一级二级缓存；真正执行SQL语句在StatementHandler执行，StatementHandler会依赖ParameterHandler进行SQL模板实参绑定，然后将Statement对象（包含SQL语句和实参）传输到数据库执行，返回ResultSet，通过ResultSetHandler将ResultSet对象转换为Java对象返回给调用方；

  • 插件：可以自定义插件来扩展Mybatis；

（3）接口层

Mybatis暴露给调用接口集合，如SqlSessionFactory、SqlSession接口，SqlSession是最为核心的接口，通过改接口可以获取Mapper代理，执行SQL语句，控制事务开关；

2、数据源和事务

（1）数据源工厂（DataSourceFactory）

  • MyBatis中的数据源工厂，采用工厂方法模式，需要扩展数据源时，只需添加对应的数据源实现和对应的工厂类即可；
  • 默认实现数据源工厂有UnpooledDataSourceFactory 和 PooledDataSourceFactory ，初始化时会创建一个DataSource空对象；
  • setProperties（）：根据传入的配置信息，对DataSource对象进行配置；
  • getDataSource（）：返回一个DataSource对象；

（2）数据源（UnpooledDataSource）

  • 核心属性：

    • driverClassLoader：加载Driver类的类加载器；
    • driverProperties：数据库连接驱动的相关配置；
    • registeredDirver：缓存所有已经注册过的数据库驱动，为线程安全；
    • defaultTransactionIsolationLevel：事务隔离级别

  • getConnection（）：调用doGetConnection方法获取数据库连接；

（3）数据源（PooledDataSource）

  • 数据库连接池优点：缓存一定数量的连接，防止被突发流量冲垮；实现连接重用，提高系统响应速度；控制连接上限，防止数据库假死；统一管理数据库连接，避免连接泄漏；

  • 数据库连接池初始化时，会初始化一定数量的数据库连接，并缓存在池中复用。当我们操作数据库时，会从池中获取数据库连接；使用完连接后，便会释放连接，重新缓存会池中；

  • PooledDataSource中维护数据库连接池的对象为PooledState，该类中维护的连接类型为PooledConnection，PooledConnection实现InvocationHandler接口，封装了Connection对象以及相关的代理对象；

  • PooledState：

    • idleConnections集合 ：存储空闲的PooledConnection对象；
    • activeConnections集合：用来存储活跃的PooledConnection对象；
①popConnection（获取连接）
  1. 检测当前是否有空闲的连接，有则直接返回连接，没有继续下一步；
  2. 检测当前连接数是否大于最大活跃连接数，小于则直接创建新连接返回，否则继续下一步；
  3. 检测当前连接中是否有超时连接，有则从活跃连接中移除该连接，继续步骤2，没有则下一步；
  4. 当前数据库连接请求阻塞，并定期执行前面三个步骤判断是否能获取连接；
②pushConnection（释放连接）
  1. 从活跃集合中删除连接，继续下一步；
  2. 检测当前连接是否可用，不可用则直接丢弃，返回，否则继续下一步；
  3. 检测当前空闲连接数是否大于最大空闲连接数，大于则直接关闭连接，否则继续下一步；
  4. 将连接重新封装，放回连接池中，唤醒所有阻塞的线程；

（4）事务接口

  • 采用工厂方法模式，TransactionFactory用于创建Transaction对象；
  • JdbcTransactionFactory 和 ManagedTransactionFactory 是 TransactionFactory 的两个实现类；
  • JdbcTransaction 并没有立即初始化数据库连接，connection 字段会被延迟初始化，具体的初始化时机是在调用 getConnection() 方法时，通过dataSource.getConnection() 方法完成初始化。
  • ManagedTransaction 的commit()、rollback() 方法都是空实现，事务的提交和回滚都是依靠容器管理的，且会初始化连接；

3、Mapper.xml与Mapper接口映射

（1）MapperRegistry

  • 核心属性：

    • config：MyBatis全局配置文件；
    • knownMappers：保存Mapper接口和MapperProxyFactory工厂对象之间的映射关系；

  • MyBatis初始化时，会读取所有Mapper.xml配置，以及扫描Mapper接口中的注解信息。之后调用addMappers（）方法，往集合knownMappers中添加<接口,MapperProxyFactory>映射关系（确保类没被注册过）；

  • 在调用Mapper接口方法操作数据库时，会调用MapperRegistry.getMapper（）方法，获取代理对象工厂类MapperProxyFactory，创建代理对象；

（2）MapperProxyFactory

创建一个MapperProxy代理对象，底层采用JDK动态代理生成代理对象，MapperProxy实现了InvocationHandler接口；

（3）MapperProxy

  • 核心属性：

    • sqlSession：记录当前MapperProcy中关联的SqlSession对象，在当前的MapperProxy代理对象中，会用该sqlSession操作数据库；
    • methodInterface：Mapper接口类型，即当前代理对象实现的接口类型；
    • methodCache：key为接口中的方法，value为方法对应的MapperMethodInvoke对象；

  • invoke（）方法：代理对象执行的入口，其中会拦截所有非Object方法，cachedInvoker会获取MapperMethod对象，并调用invoke方法执行代码逻辑；

（4）MapperMethod

  • MapperMethod是最终执行sql语句的地方，同时也记录了Mapper接口中对应的方法；
  • SqlCommon：维护了与Sql语句相关信息，为MapperMethod内部类，name记录了与sql语句的唯一标识，type维护了sql语句的操作类型（insert，update，delete，select和flush）；
  • MethodSignature：维护了Mapper接口中方法的信息；ParamNameResolver为解析方法的参数

4、MyBatis缓存

  • 采用装饰器模式，Cache为顶层接口，为具体产品类和装饰类的共同父类；
  • Cache接口中三个方法putObject（）、getObject（）、removeObject（），分别为添加缓存，获取缓存数据，删除缓存数据；
  • PerpetualCache类为Cache实现类，为被装饰类，即具体组件；
  • 除了PerpetualCache为具体实现类，其余实现Cache接口的类都为装饰类；

（1）BlockingCache

  • 对于同一个key，同一时刻，只会放行一个请求到数据库中查找数据，并将查找结果缓存到BlockingCache缓存中；
  • 每个key会关联一个CountDownLatch对象，当获取锁失败，则会调用await（）方法进行阻塞；
  • releaseLock：释放锁，会将key从map中移除，同时调用countDown（）方法唤醒其他线程

（2）FifoCache

  • 采用先进先出策略，当缓存数量达到上限时，会清理最早写入的缓存清理条目；
  • 内部维护一个keyList队列，利用LinkedLIst有序性，实现先入先出；
  • 同时有一个size字段，当缓存条目超过该数量，则会清除缓存条目；

（3）LruCache

  • 采用最近最少使用清除策略；
  • 内部维护一个LinkedHashMap，每一个条目除了包括key和value，还会包括一个before，指向前一个条目，一个after指向下一个条目；
  • LinkedHashMap中会维护一个head和tail指针，head指向第一个数据，tail指向最后一个数据，当条目被访问后，会将数据移动到tail位置，淘汰时，会将head指针指向的数据进行淘汰；
  • eldestKey：用来指向最近最少使用的key
